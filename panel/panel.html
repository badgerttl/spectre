Attack Walkthroughs
Local
none algorithm
  1. Capture a request that carries the JWT, then paste the token into Spectre and click Decode.
  2. Confirm the decoded header and payload look like the token you intend to test.
  3. Choose a none casing in Algorithm, such as none, None, or NONE.
  4. Set Attack Builder technique to alg none variants.
  5. Click Build Attack Token. Spectre rewrites the header algorithm and removes the signature.
  6. Use Copy, Send back to Cookie, or paste the token into the original request header.
  7. Replay the original request and compare status code, response body, and authorization behavior.
HS256 signing
  1. Paste the captured JWT into Spectre and click Decode.
  2. Edit the Payload claims you want to test, such as role, scope, sub, or exp.
  3. Choose Algorithm HS256.
  4. Paste the known, guessed, or cracked HMAC secret into Secret.
  5. Click Sign / Forge to create a token with a valid HMAC signature for the edited claims.
  6. Copy the output token back into the same cookie or authorization header used by the original request.
  7. Replay the request and compare the application response to the unmodified token.
kid path traversal
  1. Paste the captured JWT into Spectre and click Decode.
  2. Set Attack Builder technique to kid path traversal.
  3. Set kid to the file path you want the server to resolve, such as ../../../../dev/null.
  4. Leave HMAC secret blank when testing /dev/null. Spectre signs the token with a single null byte.
  5. If testing another readable file, paste the expected file contents into HMAC secret so the signature matches that key material.
  6. Click Build Attack Token and send the output token in the original cookie or authorization header.
  7. Replay the request and look for accepted authorization, changed identity, or server errors that reveal file lookup behavior.
Other attacks with Burp JWT Editor
Key confusion
  1. Install PortSwigger's JWT Editor extension from the BApp Store.
  2. Proxy the application through Burp, capture a request with a JWT, and send it to Repeater.
  3. Confirm the original token uses an asymmetric algorithm such as RS256, RS384, RS512, ES256, or PS256.
  4. Obtain the server's public key from a JWKS endpoint, certificate, mobile app bundle, source leak, or public metadata endpoint.
  5. Open JWT Editor's key store and create a new symmetric key using the public key bytes as the HMAC secret.
  6. In Repeater, switch to JWT Editor's token editor and change the header alg to HS256.
  7. Edit one low-risk claim first, such as extending exp or changing a test-only role.
  8. Sign the token with the HMAC key created from the public key.
  9. Send the request. A vulnerable server accepts the HS256 token even though it should only verify the original asymmetric algorithm.
Embedded JWK
  1. Send a JWT-bearing request to Burp Repeater and open the token in JWT Editor.
  2. Create a new attacker-controlled RSA key pair in JWT Editor's key store.
  3. Edit the JWT header and add a jwk parameter containing the public portion of your generated key.
  4. Set kid to match the generated key id if the tool does not add it automatically.
  5. Change the payload claim you want to validate, such as role, scope, or sub.
  6. Sign the token with the matching private key from JWT Editor.
  7. Send the request from Repeater. A vulnerable server trusts the embedded public key instead of using its own trusted key set.
jku header
  1. Create or import an attacker-controlled RSA key pair in JWT Editor.
  2. Export the public key as a JWKS document and host it at an HTTPS URL the target server can reach.
  3. Send the original JWT request to Repeater and open the token in JWT Editor.
  4. Edit the JWT header and add jku with the URL of your hosted JWKS document.
  5. Set kid to the key id inside your hosted JWKS.
  6. Edit the payload claim you want to test.
  7. Sign the token with the private key that matches the hosted JWKS public key.
  8. Send the request and check whether the server fetches and trusts the remote key URL.
x5u header
  1. Create an attacker-controlled certificate and private key, then make the certificate available at an HTTPS URL the target can reach.
  2. Import the matching private key into JWT Editor.
  3. Send the original JWT request to Repeater and open the token in JWT Editor.
  4. Edit the JWT header and add x5u with the URL of the hosted certificate or certificate chain.
  5. Set the signing algorithm to the algorithm expected by the target, commonly an RSA algorithm such as RS256.
  6. Edit one claim in the payload so you can identify whether the forged token was accepted.
  7. Sign the token with the private key that matches the hosted certificate.
  8. Send the request and check whether the server trusts the remote certificate URL.
kid SQL injection
  1. Send a JWT-bearing request to Repeater and open the token in JWT Editor.
  2. Identify whether the JWT header has a kid value. If it does not, add one for testing.
  3. Change only the kid value first, using a harmless marker such as spectre-test, then send the request and note the baseline error.
  4. Replace kid with SQL probe payloads such as ', ", or ' OR '1'='1.
  5. If the server still requires a valid signature, sign the token with the same HMAC secret or key material used by the original token.
  6. Send each request separately from Repeater and compare status codes, database errors, timing changes, and authorization behavior.
  7. If a probe changes behavior, narrow it down with one-character changes so you can distinguish SQL handling from normal unknown-key errors.
Claim tampering
  1. Send the original JWT request to Repeater and duplicate the tab so one copy stays unchanged.
  2. Open the duplicate token in JWT Editor and inspect the payload claims.
  3. Change one claim at a time, starting with low-noise fields such as exp, nbf, or aud.
  4. Move to authorization claims such as role, scope, permissions, sub, tenant, or account_id.
  5. Sign the token with the appropriate key in JWT Editor. If you do not have the key, combine this test with a discovered signing weakness such as key confusion, embedded JWK, or an accepted none algorithm.
  6. Send the modified request and compare it against the unchanged Repeater tab.
  7. Record which claim changed behavior and whether the server enforced the value cryptographically, server-side, or not at all.
